# 运营 SOP + 通知模板 + 首批 10 单检查表

## B1. 付款确认 SOP（MVP 扫码/转账）

**目的：** 防止漏单/错单/冒领；保证 48 小时交付承诺。

### 必须核对的 4 项：

1. **付款金额**：**¥499**（或你设定的价格）
2. **付款时间**：在下单后合理时间范围内
3. **付款人标识**：昵称/账号/截图一致
4. **备注**：**订单号 `ORD-xxxx`**（强烈建议要求用户备注）

### 标准操作：

- 用户发来截图/转账记录 → 运营核对上述 4 项
- 核对通过 → 调用确认接口：

```bash
curl -X POST \
  -H "Content-Type: application/json" \
  -H "x-admin-token: <ADMIN_TOKEN>" \
  -d '{"order_no":"ORD-xxxx"}' \
  http://<host>/api/orders/confirm
```

### 核对失败处理：

- **金额不符**：联系用户确认，可能是部分付款或误操作
- **时间异常**：超过 24 小时未付款，询问用户是否还需要服务
- **备注缺失**：要求用户提供订单号或重新下单
- **身份不符**：要求用户提供下单时的邮箱或手机号验证

### 确认后操作：

1. 在系统中标记订单为"已支付"
2. 通知用户可以填写深度表单
3. 记录确认时间和操作人员（可选）

## B2. 报告生成 SOP

**目的：** 确保报告质量，避免 AI 幻觉，维护品牌信誉。

### 生成前检查：

1. **订单状态**：确认订单已支付（`status = 'paid'`）
2. **提交完整性**：检查用户是否已填写所有必填问题
3. **OpenAI 配置**：确认 `OPENAI_API_KEY` 已配置

### 生成操作：

```bash
curl -X POST \
  -H "x-admin-token: <ADMIN_TOKEN>" \
  http://<host>/api/submissions/SUB-xxxx/generate
```

### 生成后检查（前 30 单强制）：

1. **内容真实性**：AI 是否编造了用户未提供的事实
2. **逻辑一致性**：判断结论是否与依据匹配
3. **格式完整性**：6 个部分是否齐全
4. **语言专业性**：是否符合咨询报告标准

### 发现问题处理：

- **严重错误**：重新生成或人工修改
- **轻微问题**：人工校对后发布
- **格式问题**：直接修正后发布

## B5. 常见问题处理

### Q1: 用户忘记备注订单号

**处理：**
1. 要求用户提供下单时间、金额、付款截图
2. 在系统中查找匹配的订单
3. 确认后手动关联订单号

### Q2: 用户付款后未收到确认

**处理：**
1. 检查系统订单状态
2. 如未确认，按 SOP 核对后确认
3. 如已确认，检查通知是否发送成功
4. 重新发送确认通知

### Q3: 报告生成失败

**处理：**
1. 检查 OpenAI API 配置和余额
2. 检查 Puppeteer 是否正常运行
3. 查看服务器日志定位错误
4. 重试生成或人工生成报告

### Q4: 用户对报告不满意

**处理：**
1. 了解具体不满意的点
2. 检查是否为 AI 幻觉或错误
3. 如为系统问题，免费重新生成
4. 如为用户理解问题，提供解释或补充说明
